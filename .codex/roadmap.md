---

## 1. Purpose

This roadmap is the **authoritative execution plan** for AI-assisted development of the Civic Intelligence Platform.

It sequences the work into approximately **16 implementation milestones** — from repository bootstrap to production deployment — so that every milestone is:

* Independently reviewable
* Independently mergeable
* Backed by explicit prerequisites
* Bounded by a clear Definition of Done
* Optimized for incremental delivery with Codex-style AI agents

The roadmap does **not** create implementation tasks or source code. It defines the boundaries, deliverables, and review gates for each milestone. Per-module task breakdown is the responsibility of the milestone's planning document.

### Authority Hierarchy

1. `AGENTS.md` (engineering constitution)
2. `docs/01`–`docs/16` (specifications — single source of truth)
3. This roadmap (sequencing and delivery plan)
4. Per-milestone implementation plans (generated by Codex at execution time)

If specifications conflict with this roadmap, the **specifications win**. If specifications conflict among themselves, stop and request clarification — never assume.

### Conventions

* **Module names** follow the `app/Modules/<Name>/` convention from §25 of `docs/03-System-Architecture.md`.
* **Backend stack:** Laravel 12, PHP 8.4, MySQL 8.4 LTS, Redis, MinIO, Sanctum, Spatie Permission (per `docs/16`).
* **Frontend stack:** React 19, TypeScript, Vite, TailwindCSS v4, TanStack Query, React Hook Form, Zod, Leaflet, Apache ECharts (per `docs/13` and `docs/16`).
* **Quality gates:** Lint → Static analysis → Unit tests → Feature tests → Integration tests → Build → Security scan (per `docs/15`).
* **Test coverage targets:** Backend ≥ 90%, Frontend ≥ 80%, Critical modules 100% (per `docs/15`).
* **Commit convention:** `type(scope): description` (per `docs/14`).
* **Complexity ratings:** **Low** (≤ 2 engineer-weeks, well-trodden patterns), **Medium** (2–4 weeks, several integrations), **High** (> 4 weeks, novel logic, multiple cross-module impacts).

### M5 — Media Pipeline & Evidence Integrity

This is the foundation that M8 (AI), M11 (department evidence viewer), and M15 (security) all depend on.

* **Modules Covered:** `Media`, `Shared` (storage abstraction, hash util, virus-scan interface).

* **Prerequisites:**
  * M1, M4
  * `docs/04-Database-Design.md` §9, §15, §25
  * `docs/05-REST-API-Specification.md` §6, §14, §22
  * `docs/06-Citizen-PWA-Specification.md` §10–12
  * `docs/11-Security-and-Anti-Fraud-Specification.md` §13–15, §32

* **Expected Deliverables:**
  * Migrations: `media`, `media_hashes` (sha256, sha512, perceptual_hash, video_fingerprint)
  * `MediaService` (upload, replace via new version — never overwrite), `HashService`, `MimeValidator`, `VirusScanService` (interface + ClamAV stub), `ThumbnailService` (queue job)
  * Storage disks configured: `media_local`, `media_minio`; MinIO bucket `cip-evidence` with versioning + object-lock
  * `POST /api/v1/reports/{id}/photos`, `POST /api/v1/reports/{id}/video` endpoints (single video per report)
  * `GET /api/v1/reports/{id}/media` (read; signed URL for playback)
  * Filename / path strategy: `evidence/{report_uuid}/{photo|video}/{uuid}.{ext}` — never reused
  * Validation: max photos 10, max size per photo, video 3–5 s enforced server-side from `duration` metadata

  * Audit logging for every upload and access
  * Background jobs: `GenerateThumbnailJob`, `ComputeHashesJob`, `ExtractVideoMetadataJob`
  * Pest feature tests for: upload happy path, file size cap, MIME rejection, magic-byte mismatch, video duration enforcement, hash determinism, signed URL TTL

* **Expected Git Commit Boundaries:**

  * `docs/04-Database-Design.md` §10
  * `docs/05-REST-API-Specification.md` §11
  * `docs/11-Security-and-Anti-Fraud-Specification.md` §28

## 4. Cross-Milestone Engineering Rules

These rules apply to every milestone and override any per-milestone shortcut.

* **Architecture wins over speed.** No business logic in controllers, components, or routes (per `docs/03` §26, `docs/14` §39).
* **Module boundaries are enforced.** No cross-module direct DB access; communicate via services or events (per `docs/03` §6).
* **Configuration over hardcoding.** Departments, categories, workflows, prompts, SLAs, AI models, connectors are all DB-driven.
* **Test first where practical.** Critical modules (auth, reports, workflow, AI pipeline, security) target 100% coverage.
* **No TODOs, no commented-out code, no debug statements** in any merged branch.
* **Every PR updates Swagger/OpenAPI and the relevant module README** when API surface changes.
* **AI must never invent APIs or columns.** If a spec is ambiguous, stop and request clarification.
* **Security defaults to on.** AuthN, AuthZ, validation, audit, rate limiting are non-negotiable on every endpoint.

Per `docs/01` §8 and `docs/16`, the following remain **explicitly out of scope** for the V1 roadmap:

* Citizen comments, up-voting, social features
* Rewards / gamification
* Live streaming
* Citizen ↔ department chat
* Automatic challan generation
* On-device AI inference
* Payment gateway
* Multilingual OCR
* IoT integrations
* Native mobile applications (PWA only in V1)
* Visual drag-and-drop workflow editor (table editor in M12, visual deferred)

Any request to add these in V1 must be raised as a scope-change review before implementation begins.

## 7. Change Control

* This roadmap is updated only when a new specification document is approved or when a milestone's prerequisites materially change.
* Per-milestone plans are the responsibility of the implementing engineer/agent and live in `plans/M<id>-*.md`.
* Any deviation from this roadmap must be documented in the PR description and approved by the Lead Solution Architect.
